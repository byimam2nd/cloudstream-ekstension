# 🔬 Verification Protocol for Fallback Strategy Implementation

## 📋 Overview

Dokumen ini menjelaskan protokol verifikasi yang **WAJIB** dilakukan sebelum mengimplementasikan fallback strategy ke production code.

**Prinsip**: "Verify First, Implement Second" - Jangan pernah implementasi tanpa verifikasi!

---

## 🎯 Tujuan Verifikasi

1. ✅ Memastikan selector baru (fallback strategy) menghasilkan output yang **SAMA** dengan selector lama
2. ✅ Mendeteksi improvement (selector baru lebih baik dari lama)
3. ✅ Mencegah regression (selector baru lebih buruk dari lama)
4. ✅ Menyediakan dokumentasi objektif untuk code review

---

## 📊 Verification Criteria

### ✅ PASS (Implementation Approved)

Selector baru di-approve jika:
- Output **SAMA** dengan selector lama, ATAU
- Output **LEBIH BAIK** (found when old failed)

**Contoh:**
```
OLD: (empty)
NEW: https://example.com/poster.jpg
Result: PASS (Better) ✓
```

### ⚠️ CHECK (Manual Review Required)

Perlu manual review jika:
- Output **BERBEDA** (tapi keduanya tidak empty)
- Format data berbeda

**Contoh:**
```
OLD: https://example.com/poster-large.jpg
NEW: https://example.com/poster-small.jpg
Result: CHECK (Manual review needed) ⚠️
```

### ❌ FAIL (Implementation Rejected)

Selector baru ditolak jika:
- Output **LEBIH BURUK** dari selector lama
- Kedua selector gagal (empty)

**Contoh:**
```
OLD: https://example.com/poster.jpg
NEW: (empty)
Result: FAIL ✗
```

## 🔬 Verification Workflow

### Step 1: Prepare Test URLs

```bash
# Contoh untuk Anichin
TEST_URLS=(
    "https://anichin.cafe/seri/carpenter-assassin/"
    "https://anichin.cafe/seri/renegade-immortal/"
    "https://anichin.cafe/seri/against-the-gods/"
)
```

**Kriteria Test URLs:**
- ✅ Minimal 3 URLs
- ✅ Include known working pages
- ✅ Include edge cases (poster missing, etc.)

### Step 2: Run Verification Script

```bash
cd /data/data/com.termux/files/home/cloudstream/cloudstream-ekstension
chmod +x scripts/verify-framework.sh
./scripts/verify-framework.sh Anichin
```

### Step 3: Review Report

```bash
# Lihat report yang di-generate
cat docs/verification/verification-anichin-*.md
```

### Step 4: Decision

Berdasarkan report:

**Jika semua PASS:**
```bash
# Proceed dengan implementasi
git add -A
git commit -m "feat(Anichin): implement fallback strategy for poster selector

Verified with: ./scripts/verify-framework.sh Anichin
Result: All tests PASS"
git push
```

**Jika ada CHECK atau FAIL:**
```bash
# Manual review dulu
# 1. Inspect HTML structure
curl -s "https://anichin.cafe/seri/carpenter-assassin/" | grep -A 5 "div.thumb"

# 2. Adjust selector jika perlu
# 3. Re-run verification
./scripts/verify-framework.sh Anichin
```

## 🧪 Test Cases per Selector Type

### Poster Selector Test

**What to verify:**
- URL valid (starts with http/https)
- URL accessible (optional: curl -I)
- Image dimensions reasonable

**Expected results:**
- Same URL as old selector, OR
- Better URL (old failed, new found)

### Title Selector Test

**What to verify:**
- Text tidak empty
- Text tidak mengandung HTML entities
- Text match dengan expected title

**Expected results:**
- Same title as old selector, OR
- Better (old failed, new found)

### Episode Selector Test

**What to verify:**
- Episode count sama
- Episode URLs valid
- Episode numbers correct

**Expected results:**
- Same episode list, OR
- More episodes found (better coverage)

### Description Selector Test

**What to verify:**
- Text tidak empty
- Text length reasonable (> 10 chars)
- Text tidak mengandung HTML tags

**Expected results:**
- Same description, OR
- Better description (longer, more complete)

## 🚨 Common Issues & Solutions

### Issue 1: Selector Mismatch

**Problem:**
```
OLD: https://example.com/image-large.jpg
NEW: https://example.com/image-small.jpg
```

**Solution:**
- Check if URL berbeda karena image optimization
- Jika new URL valid → PASS (acceptable)
- Jika new URL lower quality → Adjust selector

### Issue 2: Both Failed

**Problem:**
```
OLD: (empty)
NEW: (empty)
```

**Solution:**
- Check HTML structure (mungkin website berubah)
- Update test URL
- Add more fallback layers

### Issue 3: False Positive

**Problem:**
Selector match tapi content salah

**Solution:**
- Add content validation
- Check text length, format, etc.
- Manual inspection

## 📈 Continuous Verification

### Pre-Commit Verification

```bash
# ALWAYS run verification before commit
./scripts/verify-framework.sh [Provider]

# If PASS, then commit
git commit -m "feat(Provider): implement fallback strategy

Verified: ./scripts/verify-framework.sh [Provider]
Result: All tests PASS"
```

### Post-Deploy Verification

```bash
# After deploy, verify on production
./scripts/verify-framework.sh [Provider]

# Compare with pre-deploy report
diff docs/verification/pre-deploy-*.md docs/verification/post-deploy-*.md
```

---

## 📚 Best Practices

1. **Always verify before implement** - Jangan skip verification!
2. **Use multiple test URLs** - Minimal 3, include edge cases
3. **Save verification reports** - Untuk audit trail
4. **Re-verify after website changes** - Website structure bisa berubah
5. **Automate verification** - Integrate dengan CI/CD jika mungkin
